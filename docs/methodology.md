# Методологическая основа Prod.Swaim (для агента)

**Обязательно читать в новом чате вместе с `AGENTS.md`.**  
Это не «цитатник», а рабочий компас: дух, стиль решений и критерии правок контента. Можно и нужно выходить за рамки книг, быть любопытным, уточнять у владельца и выбирать **самую подходящую модель** под задачу — но не противоречить ядру ниже без явного решения.

Язык общения с пользователем: русский.

---

## Сводное ядро (запомнить)

1. **Исходы > фичи.** Делаем ценность для пользователя и бизнеса, не список кнопок (Cagan).
2. **Гипотезы → проверка → обучение.** Не путать движение с прогрессом (Ries).
3. **Интервью про жизнь клиента, не про нашу идею.** Похвала ≠ валидация (Fitzpatrick).
4. **Скорость обучения с реальными людьми**, сжатые циклы прототип→тест (Knapp).
5. **MVP = уже полезный и (по возможности) платящий срез продукта**, а не «сломанный черновик» и не пустышка ради галочки (Olsen + уточнение владельца).
6. **Любопытство и диалог.** Если контекста мало — спросить; если модель из другой школы лучше подходит — предложить и объяснить почему.

## 1. Marty Cagan — «Вдохновлённые» (Inspired)

### Дух и стиль
- Продукт — ремесло и ответственность за **результат для пользователя**, не «передача требований в разработку».
- Сильная product-культура: discovery + delivery; команды способны **находить** решения, а не только исполнять бэклог.
- Уважение к UX, инженерии, данным; отказ от feature factory.
- Тон правок в шаблоне: взрослый, требовательный к ясности ценности, без корпоративного тумана и без «мотивационных» пустышек.

### Принципы для правок контента
- В блоках продукта/стратегии спрашивать: *какую ценность / для кого / как поймём, что сработало?*
- Отделять **discovery** (понять проблему и решение) от **delivery** (поставить в жизнь).
- Цели и задачи вести к **outcomes**, не к «сделать 10 фич».
- Роль основателя/PM в шаблоне — владелец ценности и рисков (valuable, usable, feasible, viable), не секретарь бэклога.

### Куда в пространстве
- `Бизнес/Твой продукт`, позиционирование, цели, быстрый старт, инструкции «зачем этот раздел».

## 2. Eric Ries — «Бизнес с нуля» (Lean Startup)

### Дух
- Стартап = машина по **подтверждённому обучению** в условиях неопределённости.
- Цикл **Build → Measure → Learn**; инновационный учёт вместо vanity-метрик.
- Pivot или persevere — решение на данных и обучении, не на упрямстве или панике.

### Гипотезы
- Явно формулировать: *верим, что [клиент] имеет [проблему]; если дадим [решение], то [метрика/поведение]*.
- В шаблоне: место для гипотез, статуса проверки, вывода (подтверждена / опровергнута / уточнить).
- Эксперимент должен быть **дешёвым по времени**, но честным по сигналу.

### Пул задач (правильно)
- WIP-лимит: меньше параллели — быстрее поток обучения.
- Задачи связаны с гипотезой или outcome, не «потому что идея красивая».
- Различать: исследование / эксперимент / поставка / долг.
- Канбан-логика: To learn / Doing / Learned (или эквивалент в задачах/идеях/целях).
- Не раздувать бэклог «на всякий случай» — приоритизация по риску обучения и ценности.

### Куда в пространстве
- `Активность/Идеи`, `Задачи`, `Цели`, быстрый старт, маркетинг-эксперименты, рефлексия.

## 3. Rob Fitzpatrick — «Спроси маму» (The Mom Test)

### Дух
- Люди вежливо врут про вашу идею. Истина — в **их прошлом поведении и обязательствах**.
- Не питчить на интервью; слушать и копать конкретику.

### Правила хороших вопросов
- Про **их жизнь и работу**, не «купили бы вы наше X?»
- Про **прошлое**: «Когда в последний раз…?», «Как решали?», «Сколько времени/денег?»
- Избегать гипотетик («стали бы…», «интересно ли…»).
- Комплименты, «круто», «я бы купил» — **слабый сигнал**.
- Сильный сигнал: время, деньги, репутация, intro, предоплата, повторное использование.
- Искать workarounds и боль с цифрами; один яркий кейс лучше десяти размытых «да».

### Как понимать «что полезно»
- Полезное = подтверждённая боль + готовность менять поведение / платить.
- Идея основателя ≠ validated problem.
- В блоках клиентов/интервью давать **чеклист хороших/плохих вопросов** и красные флаги.

### Куда в пространстве
- `Бизнес/Клиенты`, интервью, отзывы, проверка идей, советы в библиотеке знаний, любые блоки «поговори с рынком».

## 4. Jake Knapp — «Спринт»

### Дух
- Сжать путь от вопроса к **прототипу и тесту с реальными людьми**.
- Скорость = скорость **обучения**, не скорость кодинга ради кодинга.
- Решение через map → варианты → выбор → prototype → test (дух 5-дневного спринта; формат можно адаптировать под соло-основателя).

### Для запусков продуктов
- Таймбокс: «к дате X есть проверяемый артефакт перед глазами пользователя».
- Прототип достаточно реалистичен, чтобы получить честную реакцию; не полировать вечность.
- Один чёткий вопрос спринта важнее десятка расплывчатых.
- После теста — явные next steps (что убрать / что усилить / что проверить ещё).

### Куда в пространстве
- Быстрый старт, идеи→эксперименты, маркетинг-кампании как спринты обучения, календарь запусков.

## 5. Dan Olsen — MVP / Lean Product (и уточнение владельца)

### Критично для Prod.Swaim
**MVP — это уже работающий срез продукта, который приносит пользу клиенту и (по возможности) деньги.**  

Не считать MVP:
- нерабочий макет «для галочки»;
- вечный beta без ценности;
- только smoke-test / фейковую кнопку **как будто это и есть продукт** (smoke-test допустим как *эксперимент*, но не подменять им определение MVP).

MVP = **полноценный продукт на узком срезе**: ясная ценность, usable, можно получить outcome (использование, удержание, оплата).

### Связка с Olsen
- Сначала problem space (кому и какая боль), потом solution space.
- Product-market fit строится на ценности UX и соответствии сегменту — не на «запустили хоть что-то».
- Урезать scope смело; урезать качество ценности — нельзя.

### Куда в пространстве
- `Твой продукт`, оффер, необходимые шаги, версии продукта, прайсинг, инструкции «что такое MVP здесь».

## Карта: методология → модули шаблона

| Модуль | Опора |
|--------|--------|
| Instructions / positioning | Cagan (зачем продукт) + обещание ценности |
| Твой продукт / оффер | Cagan + Olsen (MVP как ценный срез) |
| Клиенты / интервью | Mom Test |
| Идеи / гипотезы | Ries + Mom Test |
| Задачи / цели | Ries (WIP, outcomes) + Cagan |
| Быстрый старт / запуски | Knapp (скорость обучения) + Olsen |
| Маркетинг-эксперименты | Ries + Knapp |
| Рефлексия | Ries (learn) + Cagan (outcomes) |
| Отзывы | Mom Test (сигналы обязательств) |

При конфликте моделей: **сначала ясность ценности и честный сигнал от клиента**, потом скорость, потом полнота процесса. Спросить владельца, если спор влияет на позиционирование продукта.

## Правила поведения агента

1. Перед крупной правкой раздела — свериться с этой картой; в отчёте коротко указать, на какую школу опирались.
2. Не вставлять в шаблон «воды» и гипотетические опросы («Купили бы вы…?») без пометки, что это плохая практика.
3. Не называть MVP то, что не даёт рабочей пользы.
4. Можно предлагать практики вне этих книг (JTBD, Opportunity Solution Tree, Continuum discovery и т.д.) — с пометкой «расширение» и зачем.
5. Любопытство: задавать 1–2 точных вопроса, если без них правка будет гаданием.
6. Стиль текстов шаблона: ясный, конкретный, «вы», без инфантильности и без корпоративного канцелярита.
